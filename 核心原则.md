# 核心原则

## 五大核心原则

### 1. agent优先 (Agent-First)

**将领域任务委托给专业agent处理。**

使用场景：
- 复杂功能请求 → **planner**（规划agent）
- 代码刚写完/修改 → **code-reviewer**（审查agent）
- Bug修复或新功能 → **tdd-guide**（TDDagent）
- 架构决策 → **architect**（架构agent）
- 安全敏感代码 → **security-reviewer**（安全审查agent）

并行执行独立操作 — 同时启动多个agent。

### 2. 测试驱动 (Test-Driven)

**先写测试，再写实现，80%+ 覆盖率。**

TDD workflow：
1. 写测试（RED）— 测试应该**失败**
2. 写最小实现（GREEN）— 测试应该**通过**
3. 重构（IMPROVE）— 验证覆盖率 80%+

**覆盖率要求：80%**

必需的三种测试类型：
1. **单元测试** — 独立函数、工具、组件
2. **集成测试** — API端点、数据库操作
3. **E2E测试** — 关键用户流程

### 3. 安全第一 (Security-First)

**绝不在安全问题上妥协。**

#### 提交前必查清单

- [ ] 无硬编码密钥（API密钥、密码、Token、密钥文件路径）
- [ ] 所有用户输入已验证
- [ ] SQL注入防护（使用参数化查询）
- [ ] XSS防护（净化HTML输出）
- [ ] CSRF保护已启用
- [ ] 认证/授权已验证
- [ ] 所有端点有速率限制
- [ ] 错误消息不泄露敏感数据

#### 秘密管理

- **绝不**硬编码秘密
- 使用环境变量或秘密管理器
- 启动时验证必需的秘密
- 暴露后立即轮换

#### 发现安全问题怎么办

STOP → 使用 **security-reviewer** agent → 修复严重问题 → 轮换暴露的秘密 → 检查代码库中是否有类似问题

### 4. 不可变性 (Immutability)

**始终创建新对象，绝不修改现有对象。**

```javascript
// 错误 ❌
user.name = '新名字';
user.age = 25;

// 正确 ✓
const updatedUser = { ...user, name: '新名字', age: 25 };

// 错误 ❌
array.push(newItem);

// 正确 ✓
const newArray = [...array, newItem];

// 错误 ❌
function addItem(list, item) {
  list.push(item);
  return list;
}

// 正确 ✓
function addItem(list, item) {
  return [...list, item];
}
```

返回带有更改的新副本，而非修改原对象。

### 5. 先规划后执行 (Plan Before Execute)

**复杂功能先规划，再写代码。**

使用 `/plan` 命令：
- 明确需求
- 评估风险
- 识别依赖
- 分解为分步计划
- **等待确认后再执行**

## 代码风格

### 文件组织

- 多个小文件 > 少量大文件
- 典型 200-400 行，最多 800 行
- 按功能/领域组织，而非按类型
- 高内聚，低耦合

### 错误处理

- 每层都处理错误
- UI代码提供用户友好的消息
- 服务器端记录详细上下文
- **绝不**静默吞掉错误

### 输入验证

- 在系统边界验证所有用户输入
- 使用基于模式的验证
- 快速失败并给出明确消息
- **绝不**信任外部数据

### 代码质量清单

- 函数小（<50 行），文件专注（<800 行）
- 无深度嵌套（>4 层）
- 适当的错误处理，无硬编码值
- 可读的、良好命名的标识符

## Git workflow

### 提交格式

`<type>: <描述>`

类型：
- `feat` — 新功能
- `fix` — Bug修复
- `refactor` — 重构
- `docs` — 文档
- `test` — 测试
- `chore` — 杂务
- `perf` — 性能
- `ci` — CI/CD

### PR workflow

分析完整提交历史 → 起草全面总结 → 包含测试计划 → 用 `-u` 标志推送

---

## 性能优化

### 上下文管理

- 大重构和多文件功能**避免**使用最后 20% 的上下文窗口
- 低敏感度任务（单次编辑、文档、简单修复）可容忍更高利用率

### 构建排障

使用 **build-error-resolver** agent → 分析错误 → 逐步修复 → 每次修复后验证

## 子agent编排

| agent | 用途 | 使用时机 |
|------|------|---------|
| planner | 实施规划 | 复杂功能、重构 |
| architect | 系统设计 | 架构决策 |
| tdd-guide | 测试驱动开发 | 新功能、Bug修复 |
| code-reviewer | 代码质量 | 写完/修改代码后 |
| security-reviewer | 漏洞检测 | 提交前、敏感代码 |
| build-error-resolver | 修复构建错误 | 构建失败时 |
| e2e-runner | 端到端测试 | 关键用户流程 |
| refactor-cleaner | 死代码清理 | 代码维护 |
| doc-updater | 文档和代码图 | 更新文档 |

---

## 成功标准

- 所有测试通过，80%+ 覆盖率
- 无安全漏洞
- 代码可读且可维护
- 性能可接受
- 用户需求得到满足
