# 构建与修复

用最小、安全的变更逐步修复构建和类型错误。

## 步骤 1：检测构建系统

识别项目的构建工具并运行构建：

| 指示器 | 构建命令 |
|--------|----------|
| 带 `build` 脚本的 `package.json` | `npm run build` 或 `pnpm build` |
| `tsconfig.json`（仅 TypeScript） | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall -q .` 或 `mypy .` |

## 步骤 2：解析和分组错误

1. 运行构建命令并捕获 stderr
2. 按文件路径分组错误
3. 按依赖顺序排序（先修复导入/类型，再修复逻辑错误）
4. 计数总错误数用于进度跟踪

## 步骤 3：修复循环（一次一个错误）

每个错误：

1. **读取文件** — 使用 Read 工具查看错误上下文（错误周围 10 行）
2. **诊断** — 识别根本原因（缺失导入、错误类型、语法错误）
3. **最小修复** — 使用 Edit 工具进行最小变更以解决错误
4. **重新运行构建** — 验证错误消失且无新错误引入
5. **继续下一个** — 继续处理剩余错误

## 步骤 4：护栏

如有以下情况，停止并询问用户：
- 修复引入的错误**多于解决的错误**
- **同一错误 3 次尝试后仍存在**（可能是更深层问题）
- 修复需要**架构变更**（不只是构建修复）
- 构建错误源于**缺失依赖**（需要 `npm install`、`cargo add` 等）

## 步骤 5：摘要

显示结果：
- 已修复的错误（带文件路径）
- 剩余的错误（如有）
- 引入的新错误（应为 0）
- 未解决问题的建议后续步骤

## 恢复策略

| 情况 | 行动 |
|------|------|
| 缺失模块/导入 | 检查包是否安装；建议安装命令 |
| 类型不匹配 | 读取两种类型定义；修复更窄的类型 |
| 循环依赖 | 用导入图识别循环；建议提取 |
| 版本冲突 | 检查 `package.json` / `Cargo.toml` 版本约束 |
| 构建工具配置错误 | 读取配置文件；与工作的默认值比较 |

一次修复一个错误以确保安全。优先最小 diff 而非重构。
