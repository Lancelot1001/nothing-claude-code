# 重构清理

安全识别和移除死代码，并在每步进行测试验证。

## 步骤 1：检测死代码

根据项目类型运行分析工具：

| 工具 | 找到什么 | 命令 |
|------|--------------|---------|
| knip | 未使用的导出、文件、依赖 | `npx knip` |
| depcheck | 未使用的 npm 依赖 | `npx depcheck` |
| ts-prune | 未使用的 TypeScript 导出 | `npx ts-prune` |
| vulture | 未使用的 Python 代码 | `vulture src/` |
| deadcode | 未使用的 Go 代码 | `deadcode ./...` |
| cargo-udeps | 未使用的 Rust 依赖 | `cargo +nightly udeps` |

如果没有可用工具，使用 Grep 查找零导入的导出：
```
# 查找导出，然后检查它们是否在任何地方被导入
```

## 步骤 2：分类发现

将发现分类为安全层级：

| 层级 | 示例 | 操作 |
|------|----------|--------|
| **安全** | 未使用的工具、测试辅助函数、内部函数 | 自信删除 |
| **谨慎** | 组件、API 路由、中间件 | 验证无动态导入或外部消费者 |
| **危险** | 配置文件、入口点、类型定义 | 处理前调查 |

## 步骤 3：安全删除循环

对于每个安全项目：

1. **运行完整测试套件** — 建立基线（全部绿色）
2. **删除死代码** — 使用 Edit 工具进行精确移除
3. **重新运行测试套件** — 验证没有破坏
4. **如果测试失败** — 立即用 `git checkout -- <file>` 回滚并跳过此项目
5. **如果测试通过** — 进入下一个项目

## 步骤 4：处理谨慎项目

删除谨慎项目之前：
- 搜索动态导入：`import()`、`require()`、`__import__`
- 搜索字符串引用：路由名称、配置中的组件名称
- 检查是否从公共包 API 导出
- 验证无外部消费者（如果已发布则检查依赖者）

## 步骤 5：合并重复项

删除死代码后，寻找：
- 近似重复函数（>80% 相似）— 合并为一个
- 冗余类型定义 — 合并
- 无增值的包装函数 — 内联它们
- 无目的的重新导出 — 移除间接

## 步骤 6：摘要

报告结果：

```
死代码清理
──────────────────────────────
删除：   12 个未使用函数
          3 个未使用文件
          5 个未使用依赖
跳过：   2 个项目（测试失败）
节省：     ~450 行移除
──────────────────────────────
所有测试通过 通过：
```

## 规则

- **删除前绝不运行测试**
- **一次删除一个** — 原子变更使回滚容易
- **不确定时跳过** — 保留死代码比破坏生产更好
- **清理时不重构** — 分离关注点（先清理，稍后重构）
