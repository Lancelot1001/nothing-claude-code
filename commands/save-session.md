# 保存会话命令

捕获此会话中发生的一切 — 构建了什么、什么有效、什么失败、还剩什么 — 并写入带日期的文件，以便下一个会话可以准确地从停止的地方继续。

## 何时使用

- 关闭 Claude Code 前的工作会话结束时
- 在达到上下文限制前（先运行此命令，然后开始新会话）
- 解决了一个你想记住的复杂问题后
- 任何时候你需要将上下文移交给未来会话时

## 流程

### 步骤 1：收集上下文

写文件前，收集：

- 读取此会话中修改的所有文件（使用 git diff 或从对话中回忆）
- 回顾讨论、尝试和决定的内容
- 记录遇到的任何错误以及如何解决（或没有解决）
- 检查当前的测试/构建状态（如果相关）

### 步骤 2：创建 sessions 文件夹（如不存在）

在用户的 Claude 主目录中创建规范会话文件夹：

```bash
mkdir -p ~/.claude/session-data
```

### 步骤 3：写会话文件

创建 `~/.claude/session-data/YYYY-MM-DD-<short-id>-session.tmp`，使用今天的实际日期和满足 `session-manager.js` 中 `SESSION_FILENAME_REGEX` 规则的短 ID：

- 兼容字符：字母 `a-z` / `A-Z`、数字 `0-9`、连字符 `-`、下划线 `_`
- 兼容最小长度：1 个字符
- 新文件推荐样式：小写字母、数字和连字符，8+ 字符以避免冲突

有效示例：`abc123de`、`a1b2c3d4`、`frontend-worktree-1`、`ChezMoi_2`

新文件避免：`A`、`test_id1`、`ABC123de`

完整有效文件名示例：`2024-01-15-abc123de-session.tmp`

旧文件名格式 `YYYY-MM-DD-session.tmp` 仍然有效，但新会话文件应优先使用短 ID 形式以避免同日冲突。

### 步骤 4：填充所有章节

诚实写每个章节。不要跳过章节 — 如果某个章节确实没有内容，写"暂无"或"N/A"。不完整的文件比空章节更糟糕。

### 步骤 5：向用户展示文件

写入后，显示完整内容并询问：

```
会话已保存到 [实际解析的文件路径]

这看起来准确吗？在我们关闭前有什么要纠正或添加的吗？
```

等待确认。如有请求则进行编辑。

## 示例输出

```markdown
# Session: 2024-01-15

**Started:** ~2pm
**Last Updated:** 5:30pm
**Project:** my-app
**Topic:** 使用 httpOnly cookies 构建 JWT 认证

---

## 我们在构建什么

Next.js 应用的用户认证系统。用户通过邮箱/密码注册，
收到存储在 httpOnly cookie 中的 JWT（不是 localStorage），
受保护路由通过中间件检查有效令牌。目标是跨浏览器刷新保持会话，
而不将令牌暴露给 JavaScript。

---

## 有效的（带证据）

- **`/api/auth/register` 端点** — 确认方式：Postman POST 返回 200 带用户
  对象，行在 Supabase 仪表板可见，bcrypt 哈希正确存储
- **`lib/auth.ts` 中的 JWT 生成** — 确认方式：单元测试通过
  (`npm test -- auth.test.ts`)，jwt.io 解码令牌显示正确 payload
- **密码哈希** — 确认方式：`bcrypt.compare()` 在测试中返回 true

---

## 什么无效（及原因）

- **Next-Auth 库** — 失败原因：与我们的自定义 Prisma 适配器冲突，
  每次请求都抛出"无法在此配置中使用凭证提供商的适配器"。不值得调试 —对我们的设置太固执。
- **在 localStorage 中存储 JWT** — 失败原因：SSR 渲染在
  localStorage 可用前发生，导致每个页面加载时出现 React 水合不匹配错误。
  此方法与 Next.js SSR 基本不兼容。

---

## 尚未尝试什么

- 在登录路由响应中将 JWT 存储为 httpOnly cookie（最可能的解决方案）
- 使用 `next/headers` 中的 `cookies()` 在服务器组件中读取令牌
- 编写 middleware.ts 通过检查 cookie 存在来保护路由

---

## 做出的决策

- **httpOnly cookie 优于 localStorage** — 原因：防止 XSS 令牌盗窃，与 SSR 工作
- **自定义认证优于 Next-Auth** — 原因：Next-Auth 与我们的 Prisma 设置冲突，不值得折腾

- `cookies().set()` 在路由处理器内部还是仅在服务器操作中工作？需要验证。

在 `app/api/auth/login/route.ts` 中，生成 JWT 后，使用
`cookies().set('token', jwt, { httpOnly: true, secure: true, sameSite: 'strict' })`
将其设置为 httpOnly cookie。然后用 Postman 测试 — 响应应包含 `Set-Cookie` 头。
```

## 注意事项

- 每个会话获得自己的文件 — 绝不追加到前一会话的文件
- "什么无效"章节最关键 — 没有它，未来会话会盲目重试失败的方法
- 如果用户要求在会话中间保存（不仅是结束时），保存已知内容并明确标记进行中的项目
- 此文件旨在通过 `/resume-session` 在下一会话开始时由 Claude 读取
- 使用规范的全局会话存储：`~/.claude/session-data/`
- 任何新会话文件优先使用短 ID 文件名形式（`YYYY-MM-DD-<short-id>-session.tmp`）
