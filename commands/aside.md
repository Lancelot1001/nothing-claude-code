# Aside 命令

在任务中途提问并获得即时、聚焦的答案——然后从中断处继续。当前任务、文件和上下文不会被修改。

## 适用场景

- 当 Claude 正在工作时你突然想到一个问题，不想失去当前节奏
- 需要快速理解 Claude 正在编辑的代码
- 希望在不中断任务的情况下获得关于某个决策的第二个意见或澄清
- 在 Claude 继续之前需要理解一个错误、概念或模式
- 想问一个与当前任务无关的问题，而不启动新会话

## 使用方法

```
/aside <你的问题>
/aside 这个函数实际返回什么？
/aside 这个模式是线程安全的吗？
/aside 为什么这里用 X 而不是 Y？
/aside foo() 和 bar() 有什么区别？
/aside 我们需要担心刚才添加的 N+1 查询问题吗？
```

## 处理流程

### 步骤 1：冻结当前任务状态

在回答任何问题之前，先在脑海中记下：
- 当前活动任务是什么？（正在处理哪个文件、功能或问题）
- 调用 `/aside` 时进行到哪一步了？
- 接下来要发生什么？

在 aside 期间，**不要**读取、编辑、创建或删除任何文件。

### 步骤 2：直接回答问题

以最简洁但完整有用的方式回答问题。

- 先给答案，而不是推理过程
- 保持简短——如果需要完整解释，请在任务完成后提供深入解释
- 如果问题涉及当前正在处理的文件或代码，请精确引用（文件路径和行号）
- 如果回答需要读取文件，只读取不写入

格式如下：

```
ASIDE: [简要重述问题]

[你的回答]

— 回到任务：[正在做的一行描述]
```

### 步骤 3：恢复主任务

提供答案后，立即从任务暂停的确切位置继续活动任务。除非 aside 答案揭示了一个阻碍或需要重新考虑当前方法的原因（见边缘情况），否则不要询问是否恢复。

## 边缘情况

**未提供问题（`/aside` 后面没有任何内容）：**
回复：
```
ASIDE: 未提供问题

你想知道什么？（提出你的问题，我会回答而不会丢失当前任务上下文）

— 回到任务：[正在做的一行描述]
```

**问题揭示了当前任务的潜在问题：**
在恢复前明确标记：
```
ASIDE: [回答]

警告：注意：此回答表明当前方法存在[问题]。是否要在继续之前解决，还是按计划进行？
```
等待用户决定后再恢复。

**问题实际上是任务重定向（不是附带问题）：**
如果问题暗示要改变正在构建的内容（例如 `/aside 实际上，我们用 Redis 吧`），请澄清：
```
ASIDE: 这听起来像是方向改变，不只是附带问题。
你想：
  (a) 仅作为信息回答，保持当前计划
  (b) 暂停当前任务并改变方法
```
等待用户回答——不要假设。

**问题涉及当前打开的文件或代码：**
从实时上下文回答。如果文件在会话早期已读取，直接引用。如果没有，先读取（只读）然后用 file:line 引用回答。

**没有活动任务（调用 `/aside` 时没有任何进行中的任务）：**
仍使用标准包装器以保持响应格式一致：
```
ASIDE: [简要重述问题]

[你的回答]

— 回到任务：没有要恢复的活动任务
```

**问题需要长答案：**
简洁地给出必要答案，然后提供：
```
这是简短版本。完成后想要更深入的解释吗？[当前任务]
```

**连续多个 `/aside` 问题：**
按顺序回答每个问题。在最后一个答案之后，恢复主任务。不要在连续的 aside 中丢失任务状态。

**Aside 答案暗示需要代码更改：**
记下需要更改的内容，但不要在 aside 期间进行：
```
ASIDE: [回答]

 值得修复：[应该更改的内容]。除非你想现在解决，否则我会在当前任务后标记此问题。
```

**问题模糊或太笼统：**
问一个澄清性问题——能获取回答所需信息的最短问题。不要问多个问题。

## 示例输出

```
用户：/aside fetchWithRetry() 实际做什么？

ASIDE: fetchWithRetry() 做什么？

fetchWithRetry() (src/api/retry.ts:12) 尝试请求最多 3 次，指数退避（250ms → 500ms → 1s）。只重试 5xx 响应和网络错误——4xx 错误被视为最终结果，不会重试。

— 回到任务：重构 src/middleware/auth.ts 中的认证中间件
```

```
用户：/aside 我们采用的方法是线程安全的吗？

ASIDE: 当前方法是线程安全的吗？

不——src/cache/store.ts:34 中的共享缓存对象在无锁定的情况下被变更。
在并发请求下这是竞态条件。在单进程 Node.js 服务器中风险较低，但在使用 worker 线程或集群时会是真正的问题。

警告：注意：这可能影响我们正在构建的功能。要现在解决还是继续并在后续修复？
```

## 注意事项

- Aside 期间绝不修改文件——仅只读访问
- Aside 是对话暂停，不是新任务——原始任务必须始终恢复
- 保持答案聚焦：目标是快速解除用户的阻碍，而不是做讲座
- 如果 aside 引发更广泛的讨论，先完成当前任务，除非 aside 揭示了阻碍
- Aside 不会保存到会话文件，除非与任务结果明确相关
