# /learn-eval - 提取、评估然后保存

在写入任何skill文件之前，通过质量门禁、存储位置决策和知识放置意识扩展 `/learn`。

## 提取内容

寻找：

1. **错误解决模式** — 根本原因 + 修复 + 可重用性
2. **调试技术** — 非显而易见的步骤、工具组合
3. **变通方法** — 库怪癖、API 限制、特定版本修复
4. **项目特定模式** — 约定、架构决策、集成模式

## 流程

1. 审查会话中可提取的模式
2. 识别最有价值/可重用的见解

3. **确定保存位置：**
   - 问："这个模式在不同的项目中会有用吗？"
   - **全局**（`~/.claude/skills/learned/`）：可在 2+ 项目中使用的通用模式（bash 兼容性、LLM API 行为、调试技术等）
   - **项目**（当前项目中的 `.claude/skills/learned/`）：项目特定知识（特定配置文件的怪癖、项目特定架构决策等）
   - 不确定时，选择全局（全局 → 项目 比反过来更容易）

4. 使用此格式起草skill文件：

```markdown
---
name: pattern-name
description: "130 字符以内"
user-invocable: false
origin: auto-extracted
---

# [描述性模式名称]

**提取日期：** [日期]
**上下文：** [适用场景简要描述]

## 问题
[这个模式解决什么问题 - 要具体]

## 解决方案
[模式/技术/变通方法 - 包含代码示例]

## 使用时机
[触发条件]
```

5. **质量门禁 — 检查清单 + 综合判断**

   ### 5a. 必需检查清单（在评估草案前实际读取文件验证）

   在评估草案之前执行**所有**以下操作：

   - [ ] 按关键词 grep `~/.claude/skills/` 和相关项目 `.claude/skills/` 文件以检查内容重叠
   - [ ] 检查 MEMORY.md（项目和全局）是否有重叠
   - [ ] 考虑附加到现有skill是否足够
   - [ ] 确认这是可重用模式而非一次性修复

   ### 5b. 综合判断

   综合检查清单结果和草案质量，然后选择**一个**：

   | 判断 | 含义 | 下一步操作 |
   |---------|---------|-------------|
   | **保存** | 独特、具体、范围适当 | 进入步骤 6 |
   | **改进后保存** | 有价值但需要完善 | 列出改进 → 修订 → 重新评估（一次） |
   | **吸收到 [X]** | 应附加到现有skill | 显示目标skill和添加内容 → 步骤 6 |
   | **放弃** | 平凡、冗余或太抽象 | 解释原因并停止 |

**指南维度**（为判断提供信息，不评分）：

- **具体性 & 可操作性**：包含可立即使用的代码示例或命令
- **范围匹配**：名称、触发条件和内容一致且专注于单一模式
- **独特性**：提供现有skill未涵盖的价值（基于检查清单结果）
- **可重用性**：未来会话中存在现实触发场景

6. **判断特定的确认流程**

- **改进后保存**：展示所需改进 + 修订草案 + 更新后的检查清单/判断；修订后判断为**保存**的，在用户确认后保存，否则遵循新判断
- **保存**：展示保存路径 + 检查清单结果 + 1 行判断理由 + 完整草案 → 在用户确认后保存
- **吸收到 [X]**：展示目标路径 + 添加内容（diff 格式）+ 检查清单结果 + 判断理由 → 在用户确认后附加
- **放弃**：仅展示检查清单结果 + 理由（无需确认）

7. 保存/附加到确定的位置

## 步骤 5 的输出格式

```
### 检查清单
- [x] skills/ grep：无重叠（或：发现重叠 → 详情）
- [x] MEMORY.md：无重叠（或：发现重叠 → 详情）
- [x] 现有skill附加：新文件适当（或：应附加到 [X]）
- [x] 可重用性：已确认（或：一次性 → 放弃）

### 判断：保存 / 改进后保存 / 吸收到 [X] / 放弃

**理由：**（1-2 句话解释判断）
```

## 设计理由

此版本用基于检查清单的综合判断系统取代了之前的 5 维度数值评分方案（具体性、可操作性、范围匹配、去重、覆盖度各 1-5 分）。现代前沿模型（Opus 4.6+）具有强上下文判断能力——将丰富的定性信号强制转化为数值分数会丢失细微差别并可能产生误导性总数。综合方法让模型自然地权衡所有因素，产生更准确的保存/放弃决策，而明确的检查清单确保不会跳过任何关键检查。

## 注意事项

- 不要提取微不足道的修复（拼写错误、简单语法错误）
- 不要提取一次性问题（特定 API 中断等）
- 专注于未来会话中会节省时间的模式
- 保持skill专注——每个skill一个模式
- 判断为吸收时，附加到现有skill而非创建新文件
