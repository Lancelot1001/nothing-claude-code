# 产品镜头——构建前先思考

## 何时使用

- 开始任何功能之前——验证"为什么"
- 每周产品审查——我们在构建正确的东西吗？
- 在功能之间选择时陷入困境
- 发布之前——检查用户旅程
- 将模糊的想法转化为规格说明时

## 工作原理

### 模式 1：产品诊断

像 YC 办公时间但自动化。问出尖锐问题：

```
1. 这是给谁的？（具体的人，而不是"开发者"）
2. 痛点是什么？（量化：多久一次，有多糟糕，他们今天做什么？）
3. 为什么是现在？（发生了什么变化使这成为可能/必需的？）
4. 10 星版本是什么？（如果金钱/时间无限）
5. MVP 是什么？（证明论点的最小东西）
6. 反目标是什么？（你明确不构建什么？）
7. 你怎么知道它在起作用？（指标，而不是感觉）
```

输出：一份 `PRODUCT-BRIEF.md`，包含答案、风险和去/不去建议。

### 模式 2：创始人审查

通过创始人视角审查你当前的项目：

```
1. 阅读 README、CLAUDE.md、package.json、最近提交
2. 推断：这是试图成为什么？
3. 评分：产品-市场契合信号（0-10）
   - 使用增长轨迹
   - 留存指标（重复贡献者、回头用户）
   - 收入信号（定价页面、计费代码、Stripe 集成）
   - 竞争护城河（什么是难以复制的？）
4. 识别：能让这件事翻 10 倍的那一件事
5. 标记：你正在构建但不重要的事情
```

### 模式 3：用户旅程审计

映射实际用户体验：

```
1. 克隆/安装产品作为新用户
2. 记录每个摩擦点（令人困惑的步骤、错误、缺失文档）
3. 为每步计时
4. 与竞品入职比较
5. 评分：价值实现时间（用户多久获得他们的第一次成功？）
6. 建议：入职前 3 名修复
```

### 模式 4：功能优先级排序

当你有 10 个想法需要选择 2 个时：

```
1. 列出所有候选功能
2. 按以下方式评分：影响（1-5）× 信心（1-5）÷ 努力（1-5）
3. 按 ICE 分数排名
4. 应用约束：资金跑道、团队规模、依赖关系
5. 输出：带理由的优先级路线图
```

## 输出

所有模式都输出可操作的文档，而非论文。每个建议都有具体的下一步。

## 集成

配合使用：
- `/browser-qa` 验证用户旅程审计发现
- `/design-system audit` 进行视觉打磨评估
- `/canary-watch` 用于发布后监控
