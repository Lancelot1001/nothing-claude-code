# 上下文预算

分析 Claude Code 会话中每个加载组件的 token 开销，并提供可操作的优化建议以回收上下文空间。

## 何时使用

- 会话性能感觉变慢或输出质量下降
- 最近添加了很多skill、agent或 MCP 服务器
- 想知道实际还有多少上下文空间
- 计划添加更多组件，需要知道是否有空间
- 运行 `/context-budget` 命令（此skill支持它）

## 工作原理

### 阶段 1：清单

扫描所有组件目录并估算 token 消耗：

**agent**（`agents/*.md`）
- 计算每个文件的行数和 token 数（words × 1.3）
- 提取 `description` frontmatter 长度
- 标记：文件 >200 行（重型），描述 >30 个词（膨胀的 frontmatter）

**skill**（`skills/*/SKILL.md`）
- 计算每个 SKILL.md 的 token 数
- 标记：文件 >400 行
- 检查 `.agents/skills/` 中是否有重复副本——跳过相同副本以避免重复计算

**规则**（`rules/**/*.md`）
- 计算每个文件的 token 数
- 标记：文件 >100 行
- 检测同一语言模块中规则文件之间的内容重叠

**MCP 服务器**（`.mcp.json` 或活动的 MCP 配置）
- 计算配置的服务器数量和总工具数量
- 估算每个工具的 schema 开销约 500 token
- 标记：工具 >20 个的服务器、包装简单 CLI 命令的服务器（`gh`、`git`、`npm`、`supabase`、`vercel`）

**CLAUDE.md**（项目 + 用户级）
- 计算 CLAUDE.md 链中每个文件的 token 数
- 标记：合并总数 >300 行

### 阶段 2：分类

将每个组件分类到相应的桶中：

| 桶 | 标准 | 操作 |
|--------|----------|--------|
| **始终需要** | 在 CLAUDE.md 中被引用、支持活动命令，或匹配当前项目类型 | 保留 |
| **有时需要** | 领域特定（例如语言模式），未在 CLAUDE.md 中引用 | 考虑按需激活 |
| **很少需要** | 没有命令引用、内容重叠，或没有明显的项目匹配 | 删除或延迟加载 |

### 阶段 3：检测问题

识别以下问题模式：

- **膨胀的agent描述**——frontmatter 中 >30 个词的描述会在每次 Task 工具调用时加载
- **重型agent**——>200 行的文件会在每次生成agent时膨胀 Task 工具上下文
- **冗余组件**——复制agent逻辑的skill、复制 CLAUDE.md 的规则
- **MCP 过度订阅**——>10 个服务器，或包装免费 CLI 工具的服务器
- **CLAUDE.md 膨胀**——冗长的解释、过时的部分、应该成为规则的指令

### 阶段 4：报告

生成上下文预算报告：

```
上下文预算报告
═══════════════════════════════════════

总估算开销：~XX,XXX tokens
上下文模型：Claude Sonnet（200K 窗口）
有效可用上下文：~XXX,XXX tokens（XX%）

组件分解：
┌─────────────────┬────────┬───────────┐
│ 组件            │ 数量   │ Tokens    │
├─────────────────┼────────┼───────────┤
│ agent            │ N      │ ~X,XXX    │
│ skill            │ N      │ ~X,XXX    │
│ 规则            │ N      │ ~X,XXX    │
│ MCP 工具        │ N      │ ~XX,XXX   │
│ CLAUDE.md       │ N      │ ~X,XXX    │
└─────────────────┴────────┴───────────┘

警告：发现问题（N）：
[按 token 节省量排名]

前 3 项优化：
1. [操作] → 节省 ~X,XXX tokens
2. [操作] → 节省 ~X,XXX tokens
3. [操作] → 节省 ~X,XXX tokens

潜在节省：~XX,XXX tokens（当前开销的 XX%）
```

在详细模式下，额外输出每个文件的 token 计数、最重文件的逐行分解、重叠组件之间的特定重复行，以及每个工具 schema 大小估计的 MCP 工具列表。

## 示例

**基本审计**
```
用户：/context-budget
skill：扫描设置 → 16 个agent（12,400 tokens）、28 个skill（6,200）、87 个 MCP 工具（43,500）、2 个 CLAUDE.md（1,200）
      标记：3 个重型agent、14 个 MCP 服务器（3 个可替换 CLI）
      最大节省：移除 3 个 MCP 服务器 → -27,500 tokens（47% 开销减少）
```

**详细模式**
```
用户：/context-budget --verbose
skill：完整报告 + 每个文件的分解，显示 planner.md（213 行，1,840 tokens），
      每个工具大小的 MCP 工具列表，重复规则行并排显示
```

**扩展前检查**
```
用户：我想再添加 5 个 MCP 服务器，我有空间吗？
skill：当前开销 33% → 添加 5 个服务器（~50 个工具）将增加约 25,000 tokens → 推到 45% 开销
      建议：先移除 2 个可替换 CLI 的服务器以保持在 40% 以下
```

## 最佳实践

- **Token 估算**：对散文使用 `words × 1.3`，对代码密集型文件使用 `chars / 4`
- **MCP 是最大的杠杆**：每个工具 schema 花费约 500 tokens；一个 30 工具的服务器比你所有skill加起来还贵
- **agent描述始终加载**：即使agent从未被调用，其描述字段也存在于每次 Task 工具上下文中
- **详细模式用于调试**：在你需要精确定位导致开销的确切文件时使用，而不是常规审计
- **更改后审计**：在添加任何agent、skill或 MCP 服务器后运行，以及早发现增长
