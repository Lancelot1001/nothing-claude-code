# 代码审查标准

## 目的

代码审查确保代码在合并前的质量、安全性和可维护性。本规则定义何时及如何进行代码审查。

## 何时审查

**强制审查触发条件：**

- 编写或修改代码后
- 提交到共享分支前
- 安全敏感代码变更时（认证、支付、用户数据）
- 架构变更时
- 合并 Pull Request 前

**请求审查前的准备：**

- 所有自动化检查（CI/CD）通过
- 合并冲突已解决
- 分支与目标分支同步

## 审查清单

标记代码完成前检查：

- [ ] 代码可读且命名良好
- [ ] 函数专注（<50 行）
- [ ] 文件内聚（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 错误处理明确
- [ ] 无硬编码凭证或密钥
- [ ] 无 console.log 或调试语句
- [ ] 新功能有测试
- [ ] 测试覆盖率满足 80% 最低要求

## 安全审查触发条件

**立即停止并使用 security-reviewer agent：**

- 认证或授权代码
- 用户输入处理
- 数据库查询
- 文件系统操作
- 外部 API 调用
- 加密操作
- 支付或金融代码

## 审查严重级别

| 级别 | 含义 | 操作 |
|------|------|------|
| CRITICAL | 安全漏洞或数据丢失风险 | **BLOCK** — 合并前必须修复 |
| HIGH | 破坏功能的 bug 或重大质量问题 | **WARN** — 合并前应修复 |
| MEDIUM | 可维护性担忧 | **INFO** — 建议修复 |
| LOW | 风格不一致或小改进 | **NOTE** — 可选 |

## Agent 使用

使用这些 agents 进行代码审查：

| Agent | 职责 |
|-------|------|
| **code-reviewer** | 通用代码质量、模式、最佳实践 |
| **security-reviewer** | 安全漏洞、OWASP Top 10 |
| **typescript-reviewer** | TypeScript/JavaScript 特定问题 |
| **python-reviewer** | Python 特定问题 |
| **go-reviewer** | Go 特定问题 |
| **rust-reviewer** | Rust 特定问题 |

## 审查流程

```
1. 运行 git diff 理解变更
2. 首先检查安全清单
3. 审查代码质量清单
4. 运行相关测试
5. 验证覆盖率 >= 80%
6. 使用适当的 agent 进行详细审查
```

## 常见问题

### 安全

- 硬编码凭证（API key、密码、token）
- SQL 注入（字符串拼接构建查询）
- XSS 漏洞（未转义的用户输入）
- 路径遍历（未净化的文件路径）
- CSRF 保护缺失
- 认证绕过

### 代码质量

- 大函数（>50 行）——拆分为小函数
- 大文件（>800 行）——提取模块
- 深层嵌套（>4 层）——使用 early return
- 缺失错误处理——显式处理
- Mutation 模式——优先使用不可变操作
- 缺失测试——添加测试覆盖率

### 性能

- N+1 查询——使用 JOIN 或批处理
- 缺失分页——添加 LIMIT
- 无界查询——添加约束
- 缺失缓存——缓存昂贵操作

## 批准标准

- **Approve**：无 CRITICAL 或 HIGH 问题
- **Warning**：仅有 HIGH 问题（谨慎合并）
- **Block**：发现 CRITICAL 问题
